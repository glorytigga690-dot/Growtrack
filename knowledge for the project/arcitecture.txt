# 🏗️ **GrowTrack: Full System Architecture**

## 📐 **1. High-Level Architecture Overview**
GrowTrack follows a **3-Tier Hybrid Architecture** with role-based separation, offline-first PWA capabilities, and a dual-database strategy for optimal performance and data integrity.

```
[Client Browser / PWA] 
   │
   ├── Client Panel (Habits, Goals, Mood, Analytics, Reports, Upgrade UI)
   └── Admin Panel (User Mgmt, Subscriptions, System Metrics, Audit Logs)
   │
   ↓ HTTPS / REST API
[Application Layer: Node.js + Express]
   │
   ├── Auth & RBAC Middleware
   ├── Business Logic Controllers
   ├── Background Jobs (Reports, Sync, Subscription Checks)
   └── Payment Webhook Handlers
   │
   ↓
[Data Layer: Hybrid DB]
   ├── MySQL (ACID, Relational, Transactional)
   └── MongoDB (Flexible, Time-Series, High-Volume Logging)
```

## 🗄️ **4. Database Architecture (MySQL + MongoDB)**
Aligned with your PDF requirement for **both databases**, split by data nature:

### 🔹 **MySQL (Relational / Transactional)**
| Table | Fields | Purpose |
|-------|--------|---------|
| `users` | `id, name, email, password_hash, role, created_at, updated_at` | Core identity & RBAC |
| `subscriptions` | `id, user_id, plan_type, status, start_date, end_date, provider_ref` | Freemium tier tracking |
| `payments` | `id, user_id, amount, currency, status, transaction_id, created_at` | Billing history |
| `admin_logs` | `id, admin_id, action, target_user_id, ip, timestamp` | Audit trail & compliance |

### 🔹 **MongoDB (Document / Time-Series Friendly)**
| Collection | Fields | Purpose |
|------------|--------|---------|
| `habits` | `user_id, name, frequency, target_days, status, created_at` | Habit definitions |
| `habit_logs` | `habit_id, user_id, date, completed, synced_from` | Daily completion tracking |
| `goals` | `user_id, title, target_date, progress_percent, status, created_at` | Goal tracking |
| `mood_logs` | `user_id, date, mood_score, note, tags` | Behavioral logging |
| `report_snapshots` | `user_id, period, growth_score, consistency, mood_avg, generated_at` | Pre-computed analytics |

**Cross-DB Strategy:** All collections/collections reference `user_id` from MySQL. Application-layer joins ensure referential integrity without DB-level foreign keys.

## 🔐 **5. Security & Authentication Architecture**
| Layer | Mechanism | Purpose |
|-------|-----------|---------|
| **Password Storage** | `bcrypt` (salt rounds: 12) | Hashing before MySQL insert |
| **Session Management** | JWT (Access: 15min, Refresh: 7d) | Stateless auth, scalable |
| **Token Delivery** | `HttpOnly` cookies (recommended) or secure local storage | XSS mitigation |
| **Role-Based Access** | Middleware checks `req.user.role === 'admin'` | Panel & endpoint isolation |
| **Input Protection** | `express-validator`, DOMPurify (client), Helmet, CORS whitelist | SQLi/XSS prevention |
| **Rate Limiting** | `express-rate-limit` (100 req/hr free, 1000/hr pro) | Abuse prevention |
| **Data Rights** | `/api/v1/export`, `/api/v1/delete-account` | GDPR/privacy compliance |

## 📱 **6. PWA & Offline Sync Architecture**
```
User Action (Offline) → IndexedDB Queue → Service Worker (Background Sync) → API Retry on Online → UI Optimistic Update → Sync Status Flag (pending/synced)
```
- **Caching Strategy:** 
  - `Cache-First` for static assets (`/`, `.js`, `.css`, `manifest.json`, icons)
  - `Network-First` for `/api/*` endpoints
  - `Fallback` to `/offline.html` if disconnected
- **Sync Logic:** 
  - Queue stores `{endpoint, payload, retry_count, timestamp}`
  - `navigator.serviceWorker.ready.then(sw => sw.sync.register('sync-queue'))`
  - Backend `/api/v1/sync` endpoint processes batch uploads, validates, writes to DB

## 🛡️ **7. Admin vs Client Panel Separation**
| Aspect | Client Panel | Admin Panel |
|--------|-------------|-------------|
| **Access** | `role: 'client'` or `'admin'` | `role: 'admin'` only |
| **Route Prefix** | `/client/dashboard`, `/client/habits`, `/client/reports` | `/admin/users`, `/admin/billing`, `/admin/analytics` |
| **Features** | Tracking, mood, goals, charts, PDF export, upgrade flow | User CRUD, subscription overrides, system metrics, audit logs, content flags |
| **UI Framework** | Same vanilla JS/CSS base, different layout & components | Data tables, filters, admin-only charts, role badges |
| **Data Scope** | `WHERE user_id = current_user` | `SELECT *` with pagination & filters |

## 💳 **8. Monetization & Payment Architecture**
```
Client selects plan → Redirect to Checkout (Stripe/Razorpay Sandbox) → Payment Success → Webhook → DB: subscriptions.status = 'active' → Feature gate unlocked → UI updates
```
- **Feature Gating:** Middleware checks `user.plan` before serving Pro endpoints (e.g., `/api/v1/analytics/correlation`, `/api/v1/export/pdf`)
- **Webhook Handling:** Verifies signature, updates subscription state, logs to `payments` table, triggers confirmation email (optional)
- **Academic Demo Mode:** Environment flag `PAYMENT_MODE=sandbox` bypasses real charges but simulates full flow for viva
